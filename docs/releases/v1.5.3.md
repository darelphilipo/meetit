# v1.5.3 — Pager Spacing + Fixed Dot Position

**No API changes. No new OpenSpec items. No behavior changes.** Two visual fixes from user screenshots.

## Issue 1: My Stuff desc-pager buttons too close to the description box

**Before**: The desc-pager (`← Previous` / `5/15` / `Next →`) sat flush against the bottom of the description box. Both elements had 4px black borders, so the two borders visually merged into one thick black line (8-10px tall).

**After**: Added `margin-top: 10px` to the desc-pager so there's clear visual separation from the description box. The pager now floats naturally below the description.

Applies to all 3 My Stuff card types (RSVPs, Pitches, Events) — all 3 had the same pattern.

## Issue 2: Home card progress dots moving with title/description length

**Before**: The progress dots were placed at the END of the card header. When the title was 1 line vs 2 lines (or metadata was longer), the header height changed, pushing the dots up or down. Same for the mod card.

**After**: Moved the dots OUT of the header and into the body, just above the action row. The body is `flex: 1; min-height: 0; overflow-y: auto` — its bottom edge is always the same y-position (right above the actions). The dots now sit at a fixed visual position regardless of:
- Title length (1 line vs 2 lines)
- Description length
- Metadata wrapping
- Date badge presence (Past Event, RSVP count)

Applies to both home card and mod card (all 3 mod tabs: Pending, Published, Pitches).

## Implementation details

**For the desc-pager fix** (1-line change × 3 call sites):
```ts
// Before:
'<div id="my-stuff-desc-nav-..." style="flex-shrink:0;display:flex;..."></div>'
// After:
'<div id="my-stuff-desc-nav-..." style="flex-shrink:0;margin-top:10px;display:flex;..."></div>'
```

**For the dot positioning fix** (1-line change in 2 places):
- `renderHomeCard`: moved `<div class="card-progress" id="home-dots">` from end of `headerHtml` to end of `bodyHtml`, added `style="flex-shrink:0;margin-top:6px;"`
- `renderModCard`: same pattern, moved from headerHtml to bodyHtml

The dots remain in the same DOM tree position (just before the actions), and `updateCardDots(prefix, ...)` continues to find them via class/id selectors (no functional changes).

## What was deliberately NOT changed

- The card-shell CSS structure (header / body / actions / footer) — keeping it simple
- The dot's visual style (color, size, spacing between dots) — only position changed
- The dots' visibility logic (hidden when total ≤ 1)
- API patterns, data shapes, optimistic updates (per LEARNINGS §40, §41, §42)
