# v1.5.4 — Submit Event Form: Button Sizing + 5-Step Flow

**No API changes. No new OpenSpec items. No behavior changes.** Two visual/UX fixes from user screenshot.

## Issue 1: Back button was smaller than Submit Event button

**Before** (user screenshot): The `← Back` button was much smaller than `✓ Submit Event` — different heights, different widths. The Back text wrapped to 2 lines because the button had no min-width.

**After**: All footer buttons in the event overlay now have consistent sizing:
- `.footer-btn`: `min-width: 100px; min-height: 44px; white-space: nowrap;` (no text wrap)
- `.footer-btn-rsvp` (Submit Event): `flex: 1; min-width: 140px;` (shares the row with Back instead of taking 100% width)
- Back and Submit now appear at the same height, both single-line text

## Issue 2: Description was cramped, no separate review page

**Before** (user screenshot): Step 4 packed both the Description textarea (3 rows) AND a tiny review summary into the same page. The description "I id don't want it anymore becaluse" got cut off awkwardly because the textarea was too small and the review summary took up the remaining vertical space.

**After**: Restructured the submit-event flow to **5 steps**:

| Step | Content |
|---|---|
| 1 | Title + Category + Organizer |
| 2 | Date + Time |
| 3 | Location + Map URL |
| **4** | **Description (full page — 200px min-height textarea)** |
| **5** | **Review (full page — all details read-only, with submit button)** |

Step 4 is now dedicated to the description with a `flex:1; min-height:200px` textarea that grows to fill the page. Long descriptions no longer wrap awkwardly.

Step 5 is a full review page showing:
- Title (bold, 17px)
- Meta line: `📅 DATE at TIME · 📍 LOCATION · 🏷️ CATEGORY`
- Description in a scrollable white box (max-height: 200px)
- Helper text: "Once submitted, a mod will review and publish your event. You can edit it from My Stuff once it's published."

The review summary that was in step 4 was moved entirely to step 5 — no more cramped two-purpose page.

## Implementation

### CSS changes (`public/app.html`)
```css
.footer-btn {
  /* added: */
  min-width: 100px;
  min-height: 44px;
  white-space: nowrap;
}
.footer-btn-rsvp {
  /* changed: width: 100% → flex: 1; min-width: 140px; */
}
```

### HTML changes (`public/app.html`)
- Added `event-dot-5` to `.overlay-progress` (5 step dots)
- Added `event-step-5` as a new `.overlay-body` with review content
- Step 4 restructured: only the Description textarea, with `display:flex; flex-direction:column; flex:1; min-height:0` so the textarea grows to fill the page
- Step 5 has a new `event-review-desc-preview` element for the description text

### JS changes (`src/client/app.ts`)
- `resetEventForm`: now resets 5 steps (added "event-step-5" + "event-dot-5" to the arrays)
- `eventPrev`: now handles 4 transitions (1→2 from step 2, 2→3 from step 3, 3→4 from step 4, **4→5 from step 5**)
- `eventNext`: now handles 4 transitions (1→2, 2→3, 3→4, **4→5**). Step 4→5 validates description, populates review content, advances to step 5
- All step transitions now use `pulseDot()` for visual feedback (matches the pattern from details overlay)

The validation logic is preserved: each step validates its required fields before advancing. The submit button (which posts to `/api/submit-event`) only appears on step 5, after the user has reviewed everything.

## What was deliberately NOT changed

- API patterns, data shapes, optimistic updates, bounce guards (per LEARNINGS §40, §41, §42)
- Form validation rules (same required fields, same error toasts)
- The `submitEvent()` function itself (still reads all the same DOM values, posts the same payload)
- The 4 other step footers (my-stuff, details, mod dashboard, mod-event-details) — only the event overlay was affected
- `.event-step-4`'s old `id="event-review-title-preview"` element is now empty (only used by step 5). Could be removed, but kept for back-compat in case the submit event flow ever renders before JS is loaded.
