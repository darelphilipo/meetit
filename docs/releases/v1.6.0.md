# v1.6.0 — Pitch Form: Proposed Date & Time

Adds optional **Proposed Date** and **Proposed Time** fields to the "Pitch an Idea" form. These are suggestions from the user — not commitments — letting mods see when the pitcher imagines the event happening.

## What changed

### Form (client-side)
The pitch overlay now has 4 fields instead of 2:
- Meetup Title (required)
- Brief Description (required)
- **Proposed Date** (optional) — `📅 when you imagine it happening`
- **Proposed Time** (optional)

### Storage (server-side)
The new fields are saved to Redis on the pitch. Both are **optional** — old pitches without these fields still work, no migration needed.

### Display
The proposed date/time is shown in 2 places:
- **My Stuff → Pitches** — "💡 Proposed: 2026-06-28 at 22:38" (or "📅 Pitched: ..." if no proposed date)
- **Mod Dashboard → Pitches** — same line, bold, in the mod card header

## Scope (4 files)

1. **`public/app.html`** — 2 new form inputs (date + time) in the pitch overlay
2. **`src/shared/api.ts`** — extended `PitchFormData` with optional `proposedDate?: string; proposedTime?: string;`
3. **`src/server/server.ts`** — `onPitchIdea` reads new fields, validates with regex (`/^\d{4}-\d{2}-\d{2}$/` for date, `/^\d{2}:\d{2}$/` for time), includes them in the saved JSON
4. **`src/client/app.ts`** — 3 places:
   - `submitPitch()`: reads new fields, conditionally includes in payload
   - `renderMyPitchCard()`: shows "💡 Proposed: DATE at TIME" in header
   - `renderModCard()` pitches branch: shows proposed date/time in mod card header

## Backward compatibility

- Old pitches (pre-v1.6.0) in Redis don't have `proposedDate` / `proposedTime` fields
- Render code checks `if (item.proposedDate)` and skips the line if missing
- New pitches with empty date/time fields also skip the line
- **No migration needed** for existing data

## Server validation
```ts
if (proposedDate && !/^\d{4}-\d{2}-\d{2}$/.test(proposedDate)) return { error: "Invalid date format", status: 400 };
if (proposedTime && !/^\d{2}:\d{2}$/.test(proposedTime)) return { error: "Invalid time format", status: 400 };
```
Tight enough to prevent Redis poisoning, loose enough to accept any plausible HTML5 date/time input.

## Why optional, not required

The user said "should also have the field" — not "must require". Forcing the date would be friction for abstract pitches ("what if we did X sometime?"). The fields are there for users who want to propose timing; users who don't can leave them empty.
